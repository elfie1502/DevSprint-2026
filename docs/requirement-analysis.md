# Requirement Analysis

## Background 
The IUT cafeteria currently uses a system built to handle far fewer than the number of students there during the month of Ramadan (Iftar ordering begins at approximately 5:30PM). Every year, when Iftar ordering starts, the system fails. Orders begin piling up, the database becomes overwhelmed, and students are left staring at a spinning loader icon while their orders quietly fail. Last year, the ticketing service crashed completely and no orders were received by the kitchen; that is the problem we are attempting to solve.

We will attempt to rebuild this using a proper micro-service architecture; this means each component of the system can fail independently of the other components.

---

## Functional Requirements

### Authentication and Identity

- A student must have the ability to log into the application using his/her Student ID and password.
- The application must generate a JWT upon a successful login that expires 8 hours later.
- A student must have the ability to create a new account with his/her Student ID, Full Name, Department, and Batch.
- A student must have the ability to change their password after they successfully log in.
- There must be a pre-defined admin account with elevated permissions to allow admin access to certain features.
- The login endpoint must block a Student ID for 3 failed login attempts per minute to help prevent brute-force login attacks or scripted login attacks.

### Ordering

- Only authenticated students with a valid token can submit an order.
- A student selects a menu item, and the system checks if stock exists before processing the order.
- Each order must contain a unique idempotency key. This ensures if a student submits the same request multiple times (due to a network retry for example), the system will not create duplicate orders.
- After an order has been submitted, it will go through the following states sequentially: Pending, In Kitchen, Ready.
- A student must be able to see the real-time status of their order updates without needing to refresh the page.

### Menu and Stock

- The menu data is retrieved from the Stock Service and includes information about the Item Name, Description, Emoji, Price, and Current Quantity.
- Stock must be decremented atomically when an order is accepted.
- Simultaneous orders for the same item cannot result in over-sold items. For example, if two simultaneous requests occur and only 1 unit of stock remains, only one of the requests should succeed.
- Admin users must be able to reset stock quantities to their default values.

### Order History

- Authenticated students must have the ability to view a list of all previous orders submitted by them, ordered by most recent.

### Admin Dashboard

- Admin users must be able to see the health status of each micro-service in real-time.
- The dashboard provides real-time metrics such as the total number of orders processed, the number of failures per service, and the average latency of responses for each service.
- Admin users must be able to trigger a "chaos kill" on any single service to simulate what the system will do when a portion of services experience failures.

### Notifications

- When an order status changes, the student's browser will immediately receive the update via a persistent connection.
- Polling will not be used to retrieve updates. The front-end will connect once and listen for events.

## Non-Functional Requirements

### Performance

- Acknowledgment of an order must occur in less than 2 seconds. The actual cooking time (which takes 3 to 7 seconds) will occur in the background asynchronously.
- The Order Gateway will first check Redis Cache before querying the Stock Service. If the cache indicates that an item is out-of-stock, the request will be rejected immediately without any database queries.
- The Redis Cache has a TTL of 5 minutes and is updated whenever the full menu is queried.

### Resilience and Fault Tolerance

- Each service will run in its own Docker Container and will have a health-check endpoint.
- If the Notification Hub experiences failures, orders can still be submitted and processed. The kitchen staff can handle errors generated by the notification process without any issues.
- All services must be started correctly using a single `docker-compose up` command. Services that rely on other services will wait until those dependencies have indicated they are healthy before starting themselves.
- The idempotency key mechanism will protect against partial failures where the stock may have been decremented but the HTTP response may have never reached the client.

### Security

- Every request to submit or view an order must include a valid Bearer token. Any requests without a valid token will receive a 401 Unauthorized response.
- Any non-admin role tokens will receive a 403 Forbidden response when attempting to access admin-only endpoints.
- Passwords will be hashed using bcrypt prior to storage.


### Health Monitoring

- Each service will provide a `/health` endpoint which will return 200 when it is operational or 503 when there is an issue.
- Each service will also expose a `/metrics` endpoint containing machine readable information regarding the number of requests processed, failures experienced, and average request latency.
- The Order Gateway will aggregate health and metric information from each downstream service into a single response allowing the admin dashboard to only make calls to a single location.

| Access Level | Can Do |
| --- | --- |
| Student | Login/Register/Browse Menu/Place Order/View Live Status/View History/Change Password |
| Admin | All of the above; View Admin Dashboard/Reset Stock/Trigger Chaos Kills on services |

---

### Constraints

- The entire application must be containerized via Docker.
- Microservices will use their own service name as a hostname to communicate within the internal Docker network.
- There will be a single PostgreSQL instance as the database with three logical schemas: identity, stock, and orders.
- Redis will serve as both a cache for the Gateway and Stock Service and as the job queue backend for the Kitchen Queue (via BullMQ).
- Unit testing must validate order validation logic in the Gateway and stock decrement logic in the Stock Service. These unit tests will be executed on CI for every push to the main branch.

---

### Scope

Scope includes:

- All 5 microservices (Identity Provider, Order Gateway, Stock Service, Kitchen Queue, Notification Hub)
- Full SvelteKit frontend to cover the student journey and admin dashboard
- Local deployment configuration using Docker Compose
- Automated test execution through CI pipeline

- Outside of the scope for the required submission (not including bonus):

- Cloud-based production deployment (bonus)
- Long-term metrics storage or time-series monitoring
- Sending email/SMS notifications
- Processing payments
